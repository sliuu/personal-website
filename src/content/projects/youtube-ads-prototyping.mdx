## Project Brief

Experimenting with new designs at a large company like Google requires many layers of research before improvements can be launched. One step, that happens before running a 1% experiment, is to create a prototype of a change and test it with users.

Historically, this has been part of the work of the UX engineering team, but our team thought to create an in-house prototyping tool that everyone from UX designers to UX researchers could use to rapidly prototype and test their designs.

The first version of this tool was clunky and technical, difficult to use, and had a lot of bugs. As a result, it was still mostly UX engineers (our team) using this tool to create prototypes for other teams.

## Discovery & Research

I started with an extensive existing application audit, tracking feature inventory, current and expected CUJs, and logging all the current pain points. For example, looking at the first-time onboarding flow, the functionality for tracking state, and the functionality for creating custom components for reuse.

<figure>
<img src="/projects/prototyping-add-variables-current-flow.png" alt="Image of a flow for adding variables in the old tool" class="hero"/>
<figcaption>In this screenshot, I track the current flow for adding new variables. I look at each step the user takes and think about where the user might run into painpoints. </figcaption>
</figure>

I also did a competitive analysis, looking at other prototyping tools like ProtoPie and Framer, to get an idea of what patterns worked particularly well or did not. 

## The Solution

For the final designs, I did an aesthetic overhaul to create a more enjoyable and modern experience using the tool, created more obvious groupings to not overwhelm the user with blocks of text, made interactions more obvious, and made different sections more distinct.
